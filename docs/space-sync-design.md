# Owner-authoritative space sync design

This document defines how LocalLink spaces are represented and synchronized across local devices.

## Goals

Spaces must behave like logical shared contexts, not like automatic fan-out transport sessions.
A user can belong to a space while that space is offline, inactive locally, or temporarily unreachable.

The first version uses a single owner device per space as the source of truth.
Members cache space state locally and can control only their own local participation settings.

## Core concepts

### Space identity

Each space has a stable `space_id` generated by the owner device.
The `space_id` is the canonical identity used in invitations, membership, space-scoped messages, cached foreign spaces, local add-on settings, and update broadcasts.

### Owned space

An owned space is created on the local device.
The local device is the owner and authority for membership, invites, owner-level on/off state, and canonical metadata.

### Joined space

A joined space is accepted from an owner device.
The joined device stores a local cached copy of the space, including owner ID, name, members, revision, and current owner-published state.

### Local activation

Each device controls whether it is locally active in a space.
This is independent from direct device connections and independent from whether the device owns the space.

### Local add-ons

Each device controls which local add-ons are linked to a space on that device.
Add-on choices are not owner-authoritative and are not broadcast as canonical space state.

## Authority boundaries

### Owner-authoritative state

Only the owner can change:

- space name and canonical metadata
- owner-level space on/off state
- invitations
- accepted membership list
- removals from the space
- canonical space revision
- broadcasted space update snapshots

For now, only the owner can invite new devices.

### Member-local state

Every member device controls locally:

- whether it has accepted an invite
- whether it is locally connected/active in the space
- which local add-ons/plugins are linked to the space
- whether it leaves the space locally
- cached foreign space visibility while the owner is offline

A member leaving locally must immediately remove or disable local space keys and ignore future messages for that space unless re-invited.
If the owner is offline, the owner's membership list may remain stale until the owner later receives a leave/update event or performs reconciliation.

## Space lifecycle

### Create owned space

1. Owner creates a space locally.
2. Owner generates `space_id` and initial space revision.
3. Owner stores the space as owned.
4. Owner may turn the space on/off independently from local activation.

### Invite member

1. Owner chooses a trusted or reachable device.
2. Owner creates a pending invite for `space_id` and target device ID.
3. Invite includes enough metadata for the target device to show the invitation.
4. Invite must not silently create an accepted space on the target device.

### Accept invite

1. Target device shows the invite to the user.
2. User accepts.
3. Target stores a joined-space cache.
4. Target receives/imports the current owner-authoritative space snapshot.
5. Target can now control local activation and local add-ons for that space.
6. Owner records the device as an accepted member.

### Reject invite

1. Target declines the invite.
2. Target should not store the space as joined.
3. Owner may record declined/expired invite state, but the target is not a member.

### Owner update broadcast

When owner-authoritative state changes, the owner broadcasts a signed/encrypted space update to accepted members.
Examples:

- member accepted
- member removed
- space name changed
- owner-level on/off changed
- membership list changed
- space key epoch changed

Each update carries a monotonically increasing `revision` or equivalent ordering token.
Members apply only newer valid updates from the owner.

### Member cache while owner offline

Joined spaces remain visible locally even when the owner is offline.
The UI should show the cached space with stale/offline status.
Local add-on linking can still be edited.
Local activation may be toggled, but connecting to the space cannot succeed while the owner/state path is unavailable.

### Leave space

A member can leave at any time.
Local leave behavior is immediate:

1. mark the joined space as left or remove it from active joined spaces
2. remove or invalidate local decrypt keys for the space
3. stop local space add-ons for that space
4. ignore future space messages for that `space_id`
5. attempt to notify the owner when reachable

If the owner is offline, the owner may temporarily still list the device as a member.
The leaving device must still treat itself as gone.

## Data model direction

The current `SpaceRecord` should evolve into a model that separates owner-authoritative fields from local fields.

Suggested shape:

```rust
struct SpaceRecord {
    space_id: String,
    name: String,
    kind: SpaceKind,
    role: SpaceRole,
    owner_device_id: String,
    revision: u64,
    members: Vec<SpaceMemberRecord>,
    owner_online: bool,
    owner_enabled: bool,
    local_active: bool,
    local_addons: HashMap<String, SpaceAddonState>,
    invite_state: Option<SpaceInviteState>,
    key_epoch: u64,
}
```

Where:

```rust
enum SpaceRole {
    Owner,
    Member,
}
```

`owner_enabled` is controlled by the owner.
`local_active` is controlled by each local device.
`local_addons` is controlled by each local device.

The exact Rust structure may differ, but the separation must remain clear.

## Protocol direction

The existing space service data frame can carry space sync messages as a reserved service namespace.

Suggested service namespace:

```text
locallink.space.sync
```

Suggested message types:

- `space_invite`
- `space_invite_accept`
- `space_invite_decline`
- `space_update`
- `space_leave`
- `space_removed`
- `space_key_rotate`

All owner-authoritative updates must be authenticated as coming from the owner device.
All member-originated messages must be authenticated as coming from that member device.

## Encryption/key direction

A member that has not accepted an invite must not be able to decrypt space traffic.
A member that leaves locally must remove or disable its local space decrypt material and ignore later traffic.

A removed member should be excluded from future key epochs.
Owner-triggered removal should rotate or replace the effective space key material for remaining members.

This design does not require implementing the full key schedule in the first sync PR, but the protocol should leave room for `key_epoch` and key rotation messages.

## UI direction

Owned spaces should show:

- owner badge
- owner-level on/off control
- invite controls
- accepted members
- pending invites
- local activation control
- local add-on controls

Joined spaces should show:

- member badge
- owner device/name
- cached/offline/online state
- accept/reject pending invite state
- local activation control
- local add-on controls
- leave space action

Direct device connections remain separate from spaces.
Connecting to a space does not mean connecting to every member device.

## Implementation order

1. Extend persisted space schema for role, owner, revision, local fields, invite state, and owner fields.
2. Add owner/member validation helpers so only owners can mutate owner-authoritative fields.
3. Add invite creation and pending invite storage.
4. Add accept/decline handling and joined-space cache creation.
5. Add owner space update broadcasts with revision checks.
6. Add leave-space handling and local key invalidation placeholder.
7. Update UI to show owned vs joined spaces and pending invites.
8. Add tests for offline cache, owner-only invite, local activation, local add-ons, and leave semantics.

## Non-goals for first implementation

- multi-owner spaces
- delegated invite permissions
- cloud relay or remote server state
- automatic connection to all members
- owner-controlled member add-on settings
- guaranteed immediate owner awareness when a member leaves offline
